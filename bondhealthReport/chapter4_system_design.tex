\chapter{System Design}

\justifying
\noindent This chapter presents the comprehensive system design of the BondHealth healthcare platform. It details architectural decisions, data flow, database schema, API design, and module specifications. The design follows industry best practices for healthcare applications, ensuring scalability, security, and seamless connectivity between patients, doctors, hospital administrators, and laboratory technicians.

\section{System Architecture}

\begin{itemize}
    \item \textbf{Presentation Layer (Client Tier):} This is the user interface for all stakeholders, using server-side rendered HTML with Tailwind CSS for responsive, mobile-first design. Key advantages include fast initial page loads, improved SEO, and enhanced security by keeping business logic on the server. The interface is organized into role-specific dashboards: patient dashboards focus on appointment management, doctor dashboards prioritize clinical workflow, admin dashboards provide hospital management, and lab technician dashboards streamline test processing.

    \item \textbf{Application Layer (Business Tier):} This layer implements all business logic, validation, and security controls using Node.js with Express. It exposes RESTful API endpoints and enforces security at every request boundary. JWT-based authentication validates tokens, while Role-Based Access Control (RBAC) ensures users only perform permitted operations. A dedicated validation layer rejects malformed payloads before they reach business logic. Centralized error handling returns consistent, structured responses, and request rate limiting protects against denial-of-service conditions.

    \item \textbf{Data Layer (Database Tier):} This layer manages persistent storage using Neon PostgreSQL, a serverless solution offering automatic scaling. Connections use the serverless pooling adapter to eliminate per-request overhead. All primary keys use UUID v4. Native PostgreSQL arrays support multi-value attributes, and foreign key constraints enforce referential integrity. The serverless architecture eliminates database maintenance overhead while providing built-in high availability and automated backups.
\end{itemize}

\section{Major System Workflows}

\justifying
\noindent Data Flow Diagrams (DFDs) show how information moves through the platform, capturing inputs, transformations, and outputs at each workflow stage. This section describes three critical workflows that represent the core operational loops defining the platform's value to its users.

\begin{itemize}
    \item \textbf{Appointment Booking Flow:} Patients authenticate via JWT and search for doctors by specialization and availability. After selecting a time slot, database transactions prevent double-booking. The patient completes payment through an integrated gateway. Confirmation is dispatched via email, SMS, and in-app notifications. Both patient and doctor dashboards update immediately. This end-to-end flow typically completes in under thirty seconds.
\end{itemize}

\begin{figure}[H]
\centering
\includegraphics[width=0.95\textwidth]{images/chapter4/appointment-flow.png}
\caption{Appointment Booking Data Flow Diagram}
\label{fig:chapter4-appointment-flow}
\end{figure}
\noindent Figure~\ref{fig:chapter4-appointment-flow} presents the data flow diagram for the appointment booking workflow. The diagram shows how patient requests flow through authentication, doctor search, slot selection, payment processing, and confirmation stages. This end-to-end workflow typically completes in under thirty seconds, demonstrating the platform's efficiency in handling real-time booking operations.
\FloatBarrier

\begin{itemize}
    \item \textbf{Lab Report Processing Flow:} Lab technicians authenticate with a lab role and locate patients via UUID, name, or QR code. They select test types and enter results validated against reference ranges. Critical values trigger immediate alerts. A priority level is assigned to each report, and supporting files are virus-scanned before storage. Notifications are scaled to the priority level, ensuring urgent results are delivered without delay. This workflow reduces result delivery time from days to minutes.

    \item \textbf{Payment Processing Flow:} The payment workflow starts upon booking or post-consultation billing. The system calculates the amount due and presents payment options (card, insurance, UPI). Transactions use 3D Secure authentication. Each transaction is recorded with a unique UUID and the gateway's reference ID. All payment information is handled by PCI-compliant gateways, ensuring sensitive financial data never touches BondHealth servers directly. On success, the appointment status updates to confirmed, an invoice PDF is generated, and a receipt is emailed. Asynchronous gateway events are handled via webhook endpoints, reconciling transaction state without requiring the user to stay on the page.
\end{itemize}

\section{Database Schema Design}

\justifying
\noindent The BondHealth platform uses Neon PostgreSQL for automatic scaling and high availability. The schema defines all data and relationships using UUID primary keys, array data types, and comprehensive foreign key constraints. These design choices balance normalization requirements with query performance needs.

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{images/chapter4/schema.png}
\caption{Database Entity-Relationship Diagram}
\label{fig:schema}
\end{figure}
\justifying
\noindent The entity-relationship diagram shown in Figure~\ref{fig:schema} provides a complete visual representation of the BondHealth database structure. The diagram highlights the central role of the users table, which anchors authentication for all four system roles, and shows how foreign key relationships connect clinical records such as appointments, prescriptions, and lab reports. This schema design follows third normal form to eliminate redundancy while maintaining query performance through strategic indexes.
\FloatBarrier

\justifying
\noindent The hospitals table acts as the second major anchoring entity, grouping doctors, medicines, and lab technicians under a specific healthcare facility. The appointments table sits at the intersection of patients and doctors, linking them with scheduling, status tracking, and payment information. Clinical output tables such as lab reports and prescriptions similarly bridge patients and doctors, capturing the results of medical encounters. The payments table maintains a one-to-one relationship with appointments, recording the complete financial transaction lifecycle from intent through settlement. Operational tables like doctor leave and medicines serve narrower but equally critical purposes, supporting leave management and pharmacy inventory respectively.

\justifying
\noindent The relationships between tables are enforced through foreign key constraints with carefully chosen cascade rules. Deleting a users record cascades to the corresponding role-specific profile row, ensuring no orphaned patient or doctor records remain after account removal. By contrast, appointments and lab reports use restrict rules on their patient and doctor foreign keys, preventing accidental data loss for clinically significant records. This combination of cascade and restrict rules reflects the differing retention requirements across the schema. The diagram also shows how PostgreSQL's native array type is used for multi-value attributes such as a doctor's available weekdays and a patient's known allergies, reducing join complexity for the most frequent read queries.

\justifying
\noindent The schema follows third normal form (3NF) and uses created at and updated at timestamps for audit compliance. Strategic indexes on frequently queried columns optimize performance. Sensitive fields like password hash are never exposed via APIs. Referential integrity uses foreign key constraints with deliberate cascade rules. Deleting a users record cascades to role-specific profiles, but appointments and lab reports use restrict rules to prevent accidental loss of clinical data.

\justifying
\noindent PostgreSQL arrays store multi-value attributes like doctor availability and patient allergies, reducing join complexity for common read queries. Status fields use constrained string enumerations enforced by check constraints. All timestamps store UTC values, with the application converting to local timezones for multi-facility support.

\section{API Design}

\justifying
\noindent The BondHealth API follows RESTful best practices with resource-oriented endpoints, plural nouns, appropriate HTTP methods, and consistent JSON responses. Protected endpoints require a valid JWT token in an HTTP-only cookie, enabling horizontal scaling without server-side session storage.

\justifying
\noindent The API is organized into logical groups: authentication, patient management, doctor operations, appointments, lab workflows, prescriptions, and administration. Middleware for role verification, validation, and rate limiting is applied at the router level. List endpoints support pagination parameters, returning a standardized envelope with metadata. API versioning supports future changes.

\justifying
\noindent Error responses use a consistent JSON structure with an error code and message. HTTP status codes are standard: success codes for successful operations, validation error codes for malformed input, authentication error codes for unauthenticated requests, forbidden error codes for unauthorized access, and not found error codes for missing resources. Rate limiting protects sensitive endpoints like login and payment. Background operations return an accepted response immediately, while synchronous operations return a definitive outcome.

\justifying
\noindent All API endpoints are designed to be idempotent where appropriate, ensuring that repeated requests with the same parameters produce the same result without unintended side effects. This is particularly important for payment processing and appointment creation, where duplicate requests could otherwise lead to double billing or duplicate bookings. The API also implements conditional requests using ETags, allowing clients to cache responses and reduce unnecessary network traffic when resources have not changed.

\justifying
\noindent Request validation occurs at multiple levels within the API pipeline. Basic validation checks for required fields and data types, while business logic validation ensures operations comply with domain rules such as appointment time availability and prescription dosage limits. Validation errors return detailed messages indicating which fields failed validation and why, enabling clients to correct issues without guessing. The API also supports batch operations for common workflows, allowing clients to submit multiple related requests in a single API call, reducing latency and improving throughput.

\justifying
\noindent Authentication and authorization are enforced consistently across all protected endpoints. The JWT token embedded in the HTTP-only cookie is validated on every request, with role information extracted to determine access permissions. Session tracking maintains a record of active sessions, allowing administrators to revoke access remotely if needed. The API also logs all requests for audit purposes, capturing timestamp, user identity, endpoint accessed, and response status. These logs are retained for compliance purposes and can be queried through administrative endpoints. The table below lists core endpoints.

\section{Module-wise Detailed Design}

\justifying
\noindent This section provides design specifications for each major module. Each module encapsulates a coherent set of responsibilities and exposes a well-defined interface. The six modules are Authentication, Patient Dashboard, Doctor Dashboard, Admin Dashboard, Lab Technician interface, and Appointment Booking. This modular decomposition follows the single responsibility principle, making the codebase easier to test and maintain independently.

\subsection{Authentication Module}

\justifying
\noindent The authentication module verifies identities, manages sessions, and enforces access controls. A JWT Token Service generates signed tokens expiring after seven days, delivered via HTTP-only cookies with appropriate security flags. Server-side session state is maintained in Redis for instant revocation. Passwords are hashed with bcrypt. RBAC middleware enforces least privilege on every route. This multi-layered security approach protects against common attack vectors including XSS, CSRF, and credential theft, while the Redis-backed session store enables immediate token invalidation without waiting for natural expiry.

\subsection{Patient Dashboard Module}

\justifying
\noindent The patient dashboard provides a unified interface for managing healthcare journeys. It dynamically loads relevant data and highlights pending actions (unpaid bills, upcoming appointments, new lab reports). The dashboard includes a Profile Summary Card, Upcoming/Past Appointments sections, a Medical Records panel, Prescription Management view, Quick Action Buttons, and a Notification Center. Each component fetches only the data it needs through dedicated API endpoints, minimizing payload size and improving perceived performance. The notification center distinguishes between read and unread items, helping patients prioritize follow-up actions effectively.

\subsection{Doctor Dashboard Module}

\justifying
\noindent The doctor dashboard prioritizes clinical workflow efficiency. A header displays statistics and a status toggle for availability. Today's Schedule shows appointments with action buttons. A searchable Patient List Sidebar allows rapid access. A priority-based Lab Reports Queue ensures critical results are reviewed first. The Prescription interface includes drug interaction checking. A Leave Management Widget allows submitting and tracking leave. The dashboard also integrates a clinical notes section for documenting SOAP notes directly within the patient consultation context, reducing the need to switch between different application views during patient encounters.

\subsection{Admin Dashboard Module}

\justifying
\noindent The admin dashboard provides tools for managing hospital operations. Statistics Cards show real-time KPIs. A Quick Action Cards Grid exposes common tasks. A Doctor Management interface supports inline editing. A Medicine Inventory panel shows stock levels, low-stock alerts, and expiry dates. A Lab Technician Management list allows onboarding staff. A Leave Approval Workflow with calendar view enables efficient request disposition. All administrative actions are logged with timestamps and user identifiers, providing a complete audit trail for compliance purposes and enabling retrospective analysis of administrative decisions.

\subsection{Lab Technician Module}

\justifying
\noindent The lab technician module streamlines test processing. A multi-step Send Report Form guides result entry. A searchable Patient History Table provides prior results for trend analysis. A Statistics Dashboard displays daily report counts and turnaround times. A Quality Control Panel tracks equipment status and reagent lot information. The module also includes a barcode scanning interface for rapid patient identification using printed wristband QR codes, reducing manual data entry errors and accelerating the specimen processing workflow.

\subsection{Appointment Booking Module}

\justifying
\noindent The appointment booking module helps patients find and schedule appointments. A Doctor Search Interface with filters narrows candidates by specialization, location, and availability. A tabbed Doctor Profile View presents qualifications and fees. An Availability Calendar uses color-coded slots. A real-time Time Slot Selection grid prevents double-booking. A Booking Form captures symptoms. A Confirmation Modal handles payment and calendar integration. The module also supports waitlist enrollment for fully booked time slots, automatically notifying patients when cancellations occur and offering them the newly available slot before it becomes publicly visible.

\section{System Implementation Considerations}

\justifying
\noindent BondHealth integrates security, performance optimization, and third-party services as core implementation considerations. These elements are embedded throughout the architecture rather than treated as afterthoughts. The following subsections detail how each consideration is addressed in the implemented system.

\subsection{Security Architecture}

\justifying
\noindent BondHealth uses defense-in-depth security at every layer. Principles include Least Privilege, Secure by Default, and Complete Mediation. Security measures are applied at the network boundary, application layer, database tier, and file storage layer. The table below summarizes these measures. This layered approach ensures that compromising any single control does not grant an attacker full system access, as additional controls remain in place at other layers.

\justifying
\noindent At the network boundary, Web Application Firewall rules filter malicious traffic and DDoS protection mechanisms ensure service availability. All traffic is routed through load balancers that terminate TLS connections, offloading encryption overhead from application servers. Within the application layer, input validation occurs on both client and server sides, and all database queries use parameterized statements to prevent SQL injection. The authentication system enforces strong password policies with progressive delays after failed login attempts, and session management includes automatic timeout after inactivity.

\justifying
\noindent The database tier employs column-level encryption for sensitive fields, with database access restricted to specific service accounts. Connection strings are retrieved from secure secret management services rather than hard-coded. File storage security ensures uploaded documents are scanned for malware, with access controlled through signed URLs that expire after a configurable time window. Regular security assessments include automated vulnerability scanning, dependency checking, and periodic penetration testing. All security events are aggregated in a centralized monitoring system for real-time alerting and forensic investigation. \subsection{Database Connection Design}

\justifying
\noindent The database connection architecture uses connection pooling, strict SSL configuration, and disciplined credential management. Connection pooling maintains a cache of reusable connections, eliminating TCP handshake overhead per request. The pool is configured with appropriate minimum and maximum connections, connection timeout, idle timeout, and maximum connection lifetime parameters.

\begin{itemize}
    \item \textbf{Connection Security:} All connections use SSL mode with certificate validation. The connection string is stored in environment variables. Credentials are rotated regularly. The application uses a dedicated service account with only necessary database privileges. Query parameters are always passed separately, eliminating SQL injection.
\end{itemize}

\subsection{Third-party Integrations}

\justifying
\noindent BondHealth integrates with external services to extend functionality. All integrations use HTTPS with credentials stored exclusively in environment variables, ensuring that sensitive API keys and secrets never enter the codebase or version control. Each integration is abstracted behind a thin service wrapper, allowing the underlying provider to be swapped without changes to the calling code, which reduces vendor lock-in and simplifies maintenance when provider APIs evolve.

\justifying
\noindent Where an external service supports webhook delivery, the platform registers a dedicated endpoint to receive asynchronous event notifications. This decouples the user-facing response from the outcome of the background operation, keeping perceived response times low while ensuring reliable transaction reconciliation. Each integration is subject to the same error handling and retry conventions as internal modules, so that a transient failure in a third-party service degrades gracefully rather than propagating an unhandled exception to the user. \subsection{Performance Optimization}

\begin{itemize}
    \item \textbf{Database:} Strategic indexing and connection pooling reduce latency. Query analysis tools examine query plans. Reporting queries use read replicas. Large tables are partitioned by date.

    \item \textbf{Caching:} Redis caches frequently read data and session data. Public endpoints use CDN response caching. Expensive query results are cached to prevent redundant computation.

    \item \textbf{API \& Frontend:} List endpoints use pagination. Compression reduces response sizes significantly. HTTP/2 multiplexing enables parallel requests. Images are lazy-loaded. JavaScript bundles are code-split. Static assets come from a CDN. A service worker enables offline capability. \
    
    \item \textbf{Infrastructure:} The stateless application scales horizontally behind a load balancer. Auto-scaling groups respond to CPU and memory metrics. Performance monitoring alerts when latency exceeds thresholds.
\end{itemize}

\section{Summary}

\justifying
\noindent This chapter presented the complete system design of BondHealth, covering the three-tier architecture, major system workflows, database schema, API design, module specifications, and implementation considerations including security, database connectivity, integrations, and performance optimization. The system design was developed with primary focus on security, maintainability, scalability, and reliability to ensure that the platform can support real-world healthcare operations efficiently.

\justifying
\noindent The layered separation ensures that each architectural tier can evolve independently without affecting other components, thereby improving modularity and maintainability. The database schema provides a strong and structured foundation for managing clinical workflows, ensuring data consistency, integrity, and efficient retrieval of healthcare records. The RESTful API architecture enables standardized communication between frontend and backend components while ensuring proper role-based access control.

\justifying
\noindent Special attention has been given to security architecture to ensure that all system interactions are authenticated, authorized, and properly logged. Mechanisms such as JWT authentication, password encryption, role-based authorization, and audit logging contribute to protecting sensitive healthcare data. Performance considerations such as query optimization, connection pooling, and efficient data handling further enhance system responsiveness.

\justifying
\noindent Additionally, the system design ensures flexibility for future enhancements such as AI-based analytics, mobile application integration, advanced reporting systems, and cloud scalability. The modular design approach makes it easier to extend the system without major structural changes.

\justifying
\noindent Overall, the design provides a strong technical foundation for implementing a secure, efficient, and scalable healthcare management platform. The following chapter details the implementation of each module, translating these architectural and design decisions into a fully functional system.
