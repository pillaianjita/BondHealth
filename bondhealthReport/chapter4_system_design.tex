\chapter{System Design}

\justifying
This chapter presents the comprehensive system design of the BondHealth healthcare platform,
detailing the architectural decisions, data flow, database schema, API design, and module
specifications. The system design follows industry best practices for healthcare applications,
ensuring scalability, security, maintainability, and seamless connectivity between patients,
doctors, hospital administrators, and laboratory technicians.

% 4.1 SYSTEM ARCHITECTURE OVERVIEW
% =============================================================================

\section{System Architecture Overview}

\justifying
The BondHealth platform is built on a robust three-tier architecture that ensures
scalability, maintainability, and security while providing seamless connectivity between
all stakeholders. The architecture follows industry best practices for healthcare
applications, with clear separation of concerns between presentation, business logic,
and data persistence layers.

    \centering
    \includegraphics[width=0.9\textwidth]{images/chapter4/architecture.png}
    \caption{Three-Tier System Architecture of BondHealth Platform}

\subsection{Presentation Layer (Client Tier)}

\justifying
The presentation layer serves as the user interface through which all stakeholders interact
with the BondHealth platform. It utilizes server-side rendered HTML with Tailwind CSS
framework to deliver responsive, mobile-first design across all devices. The server-side
rendering approach offers several critical advantages for healthcare applications:

\begin{itemize}
    \item \textbf{Fast Initial Page Loads}: HTML is generated on the server and sent to the
    client, reducing client-side processing and improving time-to-interactive metrics, which
    is crucial for emergency healthcare scenarios.

    \item \textbf{Improved SEO Capabilities}: Search engines can easily crawl and index
    server-rendered content, making healthcare services more discoverable.

    \item \textbf{Enhanced Security}: Business logic remains on the server, minimizing
    client-side exposure of sensitive healthcare algorithms.

    \item \textbf{Consistent User Experience}: The interface adapts seamlessly to different
    screen sizes and devices, ensuring healthcare professionals can access patient data from
    any location.
\end{itemize}

\justifying
The user interface is organized into role-specific dashboards: patient dashboards focus on
appointment management and medical record access, doctor dashboards prioritize clinical
workflow efficiency, admin dashboards provide comprehensive hospital management capabilities,
and lab technician dashboards streamline test processing and result reporting.

\subsection{Application Layer (Business Tier)}

\justifying
The application layer forms the core of the BondHealth platform, implementing all business
logic, validation rules, workflow orchestration, and security controls. It is built using
Node.js with the Express framework, providing a lightweight, event-driven architecture ideal
for handling concurrent healthcare transactions.

\textbf{Key Components of the Application Layer:}

\begin{enumerate}
    \item \textbf{RESTful API Endpoints}: Well-defined endpoints following REST conventions
    handle all client requests, with appropriate HTTP methods and status codes ensuring
    predictable interactions.

    \item \textbf{Authentication Middleware}: JWT-based authentication verifies user identity
    for every protected request, with middleware functions that extract tokens from HTTP-only
    cookies and validate them against the active sessions store.

    \item \textbf{Role-Based Access Control (RBAC)}: Middleware functions check user roles
    against required permissions for each endpoint, ensuring patients cannot access doctor
    functions and administrators can manage hospital operations.

    \item \textbf{Request Validation Layer}: All incoming data is validated against predefined
    schemas, preventing malformed or malicious data from reaching the database.

    \item \textbf{Error Handling Middleware}: Centralized error handling captures, logs, and
    appropriately responds to exceptions, providing meaningful error messages to clients.

    \item \textbf{Request Rate Limiting}: Protection against denial-of-service attacks
    through configurable rate limiting per IP address and per user account.
\end{enumerate}

\subsection{Data Layer (Database Tier)}

\justifying
The data layer manages all persistent storage for the BondHealth platform, implementing
robust data modeling, access patterns, and security controls. Neon PostgreSQL serves as
the primary database, leveraging serverless architecture for automatic scaling based on
demand.

\textbf{Database Technology Stack Features:}

\begin{itemize}
    \item \textbf{Neon PostgreSQL}: Serverless PostgreSQL offering with built-in connection
    pooling, automatic failover, and point-in-time recovery capabilities. The serverless
    architecture scales to zero during low activity periods and instantly scales up during
    peak loads.

    \item \textbf{Connection Pooling}: Implemented through the
    \texttt{@neondatabase/serverless} package, maintaining a configurable pool of database
    connections that are reused across requests.

    \item \textbf{UUID Primary Keys}: All tables use UUID v4 as primary keys, providing
    distributed uniqueness, preventing enumeration attacks, and enabling offline record
    creation.

    \item \textbf{PostgreSQL Array Data Types}: Multi-value fields such as medical
    conditions, allergies, and doctor availability days are stored as native PostgreSQL
    arrays.

    \item \textbf{Foreign Key Constraints}: Referential integrity is enforced through
    foreign key relationships between all related tables.

    \item \textbf{SSL Encryption}: All database connections require SSL/TLS encryption,
    ensuring sensitive patient data cannot be intercepted in transit.
\end{itemize}

\section{Data Flow Diagrams}

\justifying
Data Flow Diagrams (DFDs) provide a graphical representation of how information moves
through the BondHealth platform. This section describes the three most critical workflows
in the system.

\subsection{Appointment Booking Data Flow}

\justifying
The appointment booking process enables patients to schedule consultations with doctors
based on real-time availability. Figure~\ref{fig:chapter4-appointment-flow} illustrates
this workflow.

\textbf{Detailed Process Description:}

\begin{enumerate}
    \item \textbf{Patient Authentication}: The patient provides login credentials through
    the login form. The system validates these credentials against the Users data store,
    comparing the provided password with the stored bcrypt hash. Upon successful
    authentication, the system generates a JWT token containing user identity and role
    information.

    \item \textbf{Doctor Search and Filtering}: The authenticated patient accesses the
    doctor search interface and applies filters such as specialization, hospital location,
    and availability date. The system queries the Doctors data store, applying the specified
    filters and returning matching doctor profiles.

    \item \textbf{Real-time Availability Check}: When the patient selects a specific doctor
    and date, the system queries the Appointments data store to retrieve all existing
    appointments for that doctor on the selected date. It also checks the Doctor Leave data
    store to verify the doctor is not on leave.

    \item \textbf{Slot Selection and Booking}: The patient selects an available time slot.
    The system validates that the slot remains available (preventing double-booking through
    database transactions), creates a new appointment record with status set to
    \texttt{pending}, and generates a unique appointment UUID.

    \item \textbf{Payment Processing}: For paid consultations, the system initiates the
    payment workflow, creating a payment record with status \texttt{pending} and redirecting
    to the payment gateway. Upon successful payment confirmation, the appointment status is
    updated to \texttt{confirmed}.

    \item \textbf{Notification Generation}: The system triggers the Notification Module to
    send confirmation messages through email, SMS, and in-app notifications.

    \item \textbf{Dashboard Updates}: The patient's dashboard is updated to display the
    new appointment in the ``Upcoming Appointments'' section. The doctor's dashboard is
    updated to show the new booking in their schedule.
\end{enumerate}

\subsection{Lab Report Processing Data Flow}

\justifying
Laboratory technicians use this workflow to process patient test requests, enter results,
upload supporting documents, and share reports with doctors and patients according to
configured privacy preferences. The workflow ensures that test results are accurately
recorded and securely delivered to authorized recipients.

\textbf{Detailed Process Description:}

\begin{enumerate}
    \item \textbf{Technician Authentication}: Lab technicians log into the system using
    their credentials. The authentication middleware verifies the user's role as
    \texttt{lab} and grants access to lab-specific features.

    \item \textbf{Patient Selection}: The technician accesses the patient search interface,
    which provides multiple lookup methods including search by patient UUID, name with fuzzy
    matching, or scanning QR code from patient identification band.

    \item \textbf{Test Information Entry}: The technician selects the test type from a
    predefined list of laboratory tests, each with associated normal ranges, units, and
    required input fields.

    \item \textbf{Result Entry and Validation}: The technician enters test results with
    real-time validation including range checking against normal values, unit consistency
    verification, and critical value alerts.

    \item \textbf{Priority Assignment}: The technician sets the report priority level
    (normal, urgent, critical), which determines notification urgency and dashboard
    highlighting.

    \item \textbf{File Upload}: Supporting documents can be attached to the report. The
    system validates file types, enforces size limits, and performs virus scanning before
    storage.

    \item \textbf{Sharing Preferences Configuration}: The technician specifies who can
    access the report (doctor only, patient only, or both).

    \item \textbf{Report Submission and Storage}: Upon submission, the system creates a
    new record in the Lab Reports data store, generates a unique report UUID, and triggers
    the notification workflow based on priority level.
\end{enumerate}

\subsection{Payment Processing Data Flow}

\justifying
The payment processing workflow handles all financial transactions within the BondHealth
platform, integrating with external payment gateways while maintaining comprehensive
internal records for reconciliation and audit purposes. Every transaction is recorded
with a unique payment UUID, a gateway-assigned transaction reference, and a timestamped
status trail, ensuring that the internal payment history remains consistent with the
gateway's records even in cases of network interruption or asynchronous webhook delays.

\textbf{Detailed Process Description:}

\begin{enumerate}
    \item \textbf{Payment Initiation}: The payment workflow is triggered when a patient
    books an appointment requiring upfront payment or completes a consultation with
    post-paid billing.

    \item \textbf{Amount Calculation}: The system calculates the total amount due including
    base consultation fee, applicable taxes, and any discounts or insurance adjustments.

    \item \textbf{Payment Method Selection}: The patient selects from available payment
    methods including credit/debit card, insurance billing, digital wallets, or UPI/bank
    transfer.

    \item \textbf{Payment Gateway Integration}: For card and digital wallet payments, the
    system creates a payment intent with the gateway, receives a client secret for
    front-end tokenization, and implements 3D Secure authentication if required.

    \item \textbf{Transaction Processing}: The gateway processes the payment and returns
    success, failure, or pending status. The system updates the payment record with the
    transaction ID and current status.

    \item \textbf{Post-Payment Processing}: Upon successful payment, appointment status is
    updated to \texttt{confirmed}, invoice PDF is generated and stored, receipt is emailed
    to the patient, and accounting records are updated.

    \item \textbf{Webhook Handling}: The system implements webhook endpoints for
    asynchronous gateway notifications, updating internal payment records when payments
    succeed, fail, or when disputes occur.
\end{enumerate}

\section{Database Schema Design (Neon PostgreSQL)}

\justifying
The BondHealth platform utilizes Neon PostgreSQL, a serverless PostgreSQL solution that
provides automatic scaling, high availability, and connection pooling. The database design
implements healthcare industry best practices including UUID primary keys for distributed
uniqueness, array data types for efficient multi-value storage, and comprehensive foreign
key constraints for referential integrity.

\justifying
\noindent
The Entity-Relationship diagram above illustrates the complete relational structure of
the BondHealth database. At the center of the schema is the \texttt{users} table, which
serves as the authentication anchor for all four system roles: patients, doctors, hospital
administrators, and laboratory technicians. Each role-specific table references
\texttt{users} via a foreign key on \texttt{user\_id}, ensuring a single, unified
identity per account. This centralized identity model simplifies authentication and
eliminates the need to maintain separate credential stores for each role.

\justifying
\noindent
The \texttt{hospitals} table acts as the second major anchor, grouping doctors, medicines,
and lab technicians under a specific facility. This hospital-scoped design enforces
data isolation between institutions while still allowing patients to interact with
multiple hospitals through the appointment and lab report workflows. The
\texttt{appointments} table sits at the intersection of patients and doctors, linking
both parties with scheduling, status, and payment information. The \texttt{lab\_reports}
and \texttt{prescriptions} tables similarly bridge patients and doctors, capturing the
clinical outputs of each consultation. Together, these relationships form a complete
picture of the patient journey from booking through diagnosis and treatment.

\justifying
\noindent
The \texttt{doctor\_leave} table integrates directly with the appointment availability
logic, ensuring that time slots are never offered to patients when a doctor has an
approved leave covering that date. The \texttt{medicines} table is scoped to each
hospital, enabling independent inventory management per facility, while the
\texttt{payments} table maintains a one-to-one relationship with appointments, recording
the full financial transaction lifecycle from initiation through gateway confirmation.


\justifying
\noindent
The following table provides a concise summary of each core table, its purpose, and
its primary fields as implemented in the Neon PostgreSQL schema.

\textbf{Database Design Principles:}

\begin{itemize}
    \item \textbf{Normalization}: The schema follows third normal form (3NF) to eliminate
    data redundancy while maintaining efficient query performance through strategic indexing.

    \item \textbf{Data Integrity}: Foreign key constraints with appropriate cascade rules
    ensure that related data remains consistent across all tables.

    \item \textbf{Audit Capabilities}: \texttt{created\_at} and \texttt{updated\_at}
    timestamps on all tables provide audit trails for regulatory compliance.

    \item \textbf{Performance Optimization}: Strategic indexes on frequently queried columns
    ensure rapid data retrieval even as the database scales to millions of records.

    \item \textbf{Scalability}: The use of UUID primary keys and serverless PostgreSQL
    ensures the schema can accommodate horizontal scaling without conflicts in distributed
    environments.

    \item \textbf{Security by Design}: Sensitive fields such as \texttt{password\_hash}
    are never exposed through API responses, and role-based access controls at the
    application layer ensure each user can only query data relevant to their role.
\end{itemize}

\justifying
\noindent
To support the long-term maintainability of the BondHealth platform, the database schema
is managed through a versioned migration system. Each structural change --- whether adding
a new column, introducing a junction table, or modifying a constraint --- is captured as an
incremental migration script, allowing the schema to evolve in a controlled and reversible
manner across development, staging, and production environments. This ensures that no
ad-hoc alterations are made directly to the production database, reducing the risk of
inconsistency between environments.

\noindent
The migration pipeline also plays a critical role in onboarding new hospital instances.
Since each facility operates within the same shared schema but maintains logically isolated
data through foreign key scoping, new hospitals can be provisioned without structural
changes. Seed scripts initialise default roles, department categories, and configuration
records for each new facility, ensuring a consistent baseline across all institutions from
the moment they are onboarded.

\section{API Design (RESTful Endpoints)}

\justifying
\noindent
The BondHealth platform implements a comprehensive RESTful API architecture that follows
industry best practices for web service design. All endpoints adhere to REST conventions
with appropriate HTTP methods, consistent resource naming, and standardised response
formats. The API serves as the communication backbone between the presentation layer and
the business logic, handling all data exchange between the four user roles and the server.

\noindent
Each endpoint is designed to perform a single, well-defined operation, making the API
predictable and easy to consume by both the web interface and future mobile clients. Error
responses follow a consistent JSON structure containing an error code, human-readable
message, and optional field-level validation details, enabling the frontend to provide
meaningful feedback to users. All protected endpoints require a valid JWT token delivered
via HTTP-only cookie, ensuring that authentication state is never exposed to client-side
JavaScript and remains resistant to cross-site scripting attacks.

\noindent
The API is organised into logical resource groups that mirror the core domain entities of
the platform. Separate route modules handle authentication, patient management, doctor
operations, appointment scheduling, laboratory workflows, prescription handling, and
administrative functions. This modular structure not only improves code maintainability but
also allows middleware to be applied selectively at the router level, meaning that role
verification, request validation, and rate limiting can each be scoped precisely to the
endpoints that require them.

\noindent
Request validation is enforced at the boundary of each endpoint using a dedicated schema
validation layer. Incoming payloads are checked for type correctness, required field
presence, and domain-specific constraints --- such as ensuring appointment times fall within
a doctor's available schedule --- before any business logic is executed. This fail-fast
approach prevents malformed data from propagating into the service layer or the database,
reducing the surface area for both bugs and security vulnerabilities.

\noindent
The API also implements consistent pagination across all list-returning endpoints. Rather
than returning unbounded result sets, every collection endpoint accepts \texttt{page} and
\texttt{limit} query parameters and returns a standardised envelope containing the data
array, total record count, and current page metadata. This design ensures predictable
memory consumption on both the server and the client, and lays the groundwork for
cursor-based pagination as data volumes grow. Caching headers are applied selectively to
read-heavy endpoints such as hospital listings and doctor availability, reducing redundant
database queries under high concurrency without sacrificing data freshness for
time-sensitive resources such as appointment slots.

\textbf{API Design Principles:}

\begin{itemize}
    \item \textbf{Resource-Oriented}: Endpoints are organized around resources (patients,
    doctors, appointments) rather than actions.
    \item \textbf{Consistent Naming}: Plural nouns for collections and singular for specific
    resources.
    \item \textbf{HTTP Methods}: GET for retrieval, POST for creation, PUT for updates,
    DELETE for removal.
    \item \textbf{Status Codes}: Appropriate HTTP status codes provide clear feedback about
    operation outcomes.
    \item \textbf{Versioning}: API versioning through URL path (/api/v1/\ldots) supports
    future changes.
    \item \textbf{Authentication}: JWT tokens in HTTP-only cookies for all protected
    endpoints.
\end{itemize}

\section{Module-wise Detailed Design}

\justifying
This section provides comprehensive design specifications for each major module of the
BondHealth platform, describing their internal architecture, component structure, and
interactions with other modules.

\subsection{Authentication Module}

\justifying
The authentication module is the security gateway for the BondHealth platform, responsible
for verifying user identities, managing sessions, and enforcing access controls.

\textbf{Core Components:}

\begin{enumerate}
    \item \textbf{JWT Token Service}: Generates and validates JSON Web Tokens with user ID,
    username, role, and session ID in the payload. Tokens are signed with HS256 algorithm
    and expire after 7 days.

    \item \textbf{HTTP-Only Cookie Management}: Tokens are stored in HTTP-only cookies with
    \texttt{HttpOnly}, \texttt{Secure}, and \texttt{SameSite=Strict} flags, preventing XSS
    and CSRF attacks.

    \item \textbf{Server-Side Session Store}: Maintains active sessions in memory with Redis
    backing for distributed deployments, enabling forced logout of specific sessions.

    \item \textbf{Password Hashing Service}: Implements bcrypt with 10 salt rounds,
    providing secure password storage with constant-time comparison to prevent timing
    attacks.

    \item \textbf{Role-Based Access Control (RBAC) Middleware}: Checks user roles against
    required permissions for each endpoint, implementing the principle of least privilege.
\end{enumerate}

\subsection{Patient Dashboard Module}

\justifying
The patient dashboard provides a comprehensive, intuitive interface for patients to manage
all aspects of their healthcare journey, aggregating information from multiple modules into
a single, unified view. It is designed with simplicity in mind, ensuring that patients of
all technical backgrounds can navigate their medical records, schedule appointments, and
communicate with their doctors without friction. The dashboard dynamically loads only the
data relevant to the logged-in patient, ensuring both performance efficiency and strict
data privacy. Real-time indicators highlight pending actions such as unpaid bills, upcoming
appointments within the next 24 hours, and newly uploaded lab reports, keeping patients
informed and engaged with their own healthcare management.

\textbf{Component Architecture:}

\begin{itemize}
    \item \textbf{Profile Summary Card}: Displays patient name, blood type, age, emergency
    contact information, and insurance summary with quick edit functionality.

    \item \textbf{Upcoming Appointments Section}: Lists upcoming appointments with date,
    time, doctor name, and action buttons for rescheduling or cancellation.

    \item \textbf{Past Appointments Section}: Paginated list of completed appointments with
    expandable details and option to book follow-up appointments.

    \item \textbf{Medical Records Section}: Tabbed interface for lab reports, prescriptions,
    imaging, and vaccinations with search and filter capabilities.

    \item \textbf{Prescription Management Area}: Active prescriptions with medicine details,
    refill status, and one-click refill requests.

    \item \textbf{Quick Action Buttons}: Book New Appointment, Message My Doctor, View Lab
    Reports, Pay Outstanding Bills.

    \item \textbf{Notification Center}: Recent notifications with read/unread status,
    appointment reminders, and lab report alerts.
\end{itemize}

\subsection{Doctor Dashboard Module}

\justifying
The doctor dashboard is designed for clinical workflow efficiency, providing quick access
to today's schedule, patient information, and pending tasks.

\textbf{Component Architecture:}

\begin{itemize}
    \item \textbf{Dashboard Header}: Welcome message with doctor name, current date/time,
    quick stats, and status toggle.

    \item \textbf{Today's Schedule Component}: Chronological list of today's appointments
    with patient information, status indicators, and action buttons (Start Consultation,
    View History, Message Patient).

    \item \textbf{Patient List Sidebar}: Searchable list of all patients under the doctor's
    care with quick filters for recent, frequent, and new patients.

    \item \textbf{Lab Reports Queue}: Priority-based organization of pending lab reports
    with critical, urgent, and normal categories.

    \item \textbf{Prescription Writing Interface}: Medicine search with auto-complete,
    dosage templates, frequency picker, and drug interaction checker.

    \item \textbf{Leave Management Widget}: Upcoming leave summary, leave application form,
    and approval status tracking.
\end{itemize}

\subsection{Admin Dashboard Module}

\justifying
The admin dashboard provides hospital administrators with comprehensive tools for managing
the entire healthcare facility operations.

\textbf{Component Architecture:}

\begin{itemize}
    \item \textbf{Statistics Cards Row}: Displays total doctors, today's appointments,
    pending approvals, inventory status, and revenue metrics.

    \item \textbf{Quick Action Cards Grid}: Add New Doctor, Add Medicine to Inventory, Add
    Lab Technician, View All Appointments, Generate Reports.

    \item \textbf{Doctor Management Interface}: Searchable table of all doctors with inline
    editing for status, department, and fees, plus credential verification workflow.

    \item \textbf{Medicine Inventory Management}: Real-time stock levels with low stock
    alerts, expiry tracking, and batch management.

    \item \textbf{Lab Technician Management}: List of all lab technicians with roles,
    certifications, and workload distribution.

    \item \textbf{Leave Approval Workflow}: Calendar view showing all leave requests with
    quick approve/reject buttons and coverage analysis.
\end{itemize}

\subsection{Lab Technician Module}

\justifying
The laboratory technician module streamlines the complete workflow of test processing,

\section{Security Architecture}

\justifying
The BondHealth platform implements a comprehensive, defense-in-depth security architecture
to protect sensitive healthcare data and ensure regulatory compliance. Security is treated
not as a peripheral concern but as a foundational requirement embedded at every layer of
the system, from database constraints and API middleware through to client-side token
handling and session management.

\textbf{Security Principles:}
\begin{itemize}
    \item \textbf{Defense in Depth:} Multiple layers of security controls ensure that if
    one layer fails, others continue to provide protection.

    \item \textbf{Least Privilege:} Users and services have only the minimum permissions
    necessary to perform their functions, limiting the potential blast radius of any
    compromised account or service.

    \item \textbf{Secure by Default:} Default configurations are secure, requiring explicit
    actions to reduce security controls rather than to enable them, ensuring that
    misconfiguration cannot silently weaken the system's protective posture.

    \item \textbf{Complete Mediation:} Every access attempt is checked for authorisation,
    not just the first, ensuring that no request can bypass role verification through
    session replay, token reuse, or direct resource access via guessed identifiers.
\end{itemize}

\section{Database Connection Design (Neon PostgreSQL)}

\justifying
The database connection architecture is designed for optimal performance, reliability,
and security when connecting the Node.js application layer to Neon PostgreSQL.

\subsection{Connection Pooling Architecture}

\justifying
Connection pooling maintains a cache of database connections that can be reused across
multiple client requests, eliminating the overhead of establishing a new connection for
each database operation. In a healthcare platform like BondHealth, where concurrent
requests from patients, doctors, administrators, and lab technicians are common, efficient
connection management is critical to maintaining responsive performance. Without pooling,
each API request would require a full TCP handshake and authentication round-trip to the
Neon PostgreSQL server, adding significant latency to every database interaction. The
pool maintains a configurable minimum number of ready connections at all times, ensuring
that even sudden traffic spikes are handled gracefully without exhausting database
resources. Failed or stale connections are automatically detected and replaced, providing
resilience without any manual intervention.

\textbf{Pool Configuration:}
\begin{itemize}
    \item \textbf{Min Pool Size}: 5 connections to handle sudden spikes.
    \item \textbf{Max Pool Size}: 20 connections to limit database load.
    \item \textbf{Connection Timeout}: 10 seconds to wait for a connection.
    \item \textbf{Idle Timeout}: 30 seconds to close idle connections.
    \item \textbf{Max Lifetime}: 30 minutes to recycle stale connections.
\end{itemize}

\textbf{Pool Benefits:}
\begin{itemize}
    \item Reduced latency (no connection setup overhead).
    \item Better resource utilization (controlled number of connections).
    \item Automatic recovery (replaces failed connections).
    \item Load smoothing (handles traffic spikes gracefully).
\end{itemize}

\begin{lstlisting}[language=JavaScript,
    caption={Database Connection Configuration (config/database.js)},
    label={lst:dbconfig},
    breaklines=true,
    columns=flexible,
    keepspaces=true]
const { Pool } = require('@neondatabase/serverless');

const pool = new Pool({
    connectionString: process.env.DATABASE_URL,
    ssl: { require: true, rejectUnauthorized: true },
    max: 20,
    idleTimeoutMillis: 30000,
    connectionTimeoutMillis: 10000,
});

const query = async (text, params) => {
    const start = Date.now();
    try {
        const res = await pool.query(text, params);
        const duration = Date.now() - start;
        if (duration > 100) {
            console.warn('Slow query detected:', { text, duration });
        }
        return res;
    } catch (error) {
        console.error('Database query error:', error);
        throw error;
    }
};
\end{lstlisting}

\subsection{Connection Security}

\textbf{SSL/TLS Configuration:}
\begin{itemize}
    \item SSL mode: \texttt{require} (enforces encryption).
    \item Certificate validation: Enabled (prevents MITM attacks).
    \item Minimum TLS version: TLSv1.2.
\end{itemize}

\textbf{Credential Management:}
\begin{itemize}
    \item Connection string stored in environment variables.
    \item Never committed to version control.
    \item Rotated quarterly or after any security incident.
\end{itemize}

\section{Performance Optimization}

\justifying
The BondHealth platform implements multiple optimization strategies to ensure responsive
user experiences even under high load.

\textbf{Database Optimization:}
\begin{itemize}
    \item Database indexing on frequently queried columns significantly reduces query
    execution times.
    \item Connection pooling minimizes database connection overhead.
    \item Query optimization through EXPLAIN analysis and index tuning.
    \item Read replicas for reporting queries to offload primary database.
    \item Partitioning for large tables (appointments by date, logs by month).
\end{itemize}

\textbf{Caching Strategy:}
\begin{itemize}
    \item Redis caching for frequently accessed data such as doctor lists and hospital
    information.
    \item Session data stored in Redis for sub-millisecond access.
    \item Response caching for public endpoints with CDN.
    \item Database query caching for expensive but static queries.
\end{itemize}

\textbf{API Optimization:}
\begin{itemize}
    \item Pagination for all list endpoints ensuring response sizes remain manageable.
    \item Field selection to allow clients to request only needed data.
    \item Compression (gzip/brotli) for API responses reduces bandwidth by 70--80\%.
    \item HTTP/2 multiplexing for parallel requests.
    \item Batch endpoints for mobile clients to reduce round trips.
\end{itemize}

\textbf{Frontend Optimization:}
\begin{itemize}
    \item Lazy loading for images and file attachments.
    \item Code splitting for JavaScript bundles.
    \item Asset minification and bundling.
    \item CDN for static assets with edge caching.
    \item Service workers for offline capability and caching.
\end{itemize}

\textbf{Infrastructure Scaling:}
\begin{itemize}
    \item Horizontal scaling through stateless application design.
    \item Load balancing with NGINX/HAProxy across instances.
    \item Auto-scaling groups based on CPU and memory metrics.
    \item Database read replicas to distribute query load.
    \item CDN for static assets reduces origin load by 90\%.
\end{itemize}

\textbf{Monitoring and Alerting:}
\begin{itemize}
    \item Real-time performance monitoring with DataDog/New Relic.
    \item Alerting on performance degradation (p95 latency $>$ 500ms).
    \item Log aggregation for slow query analysis.
    \item Synthetic monitoring for critical user journeys.
\end{itemize}
