\chapter{Implementation}
\label{ch:implementation}

\justifying
\noindent This chapter details the implementation of the BondHealth hospital management system, describing each module's routing logic, database interactions, and integration with the broader application. The implementation follows a layered architecture: Express.js handles HTTP routing, Neon PostgreSQL serves as the persistent store, and server-rendered EJS templates present the user interface. All code is organised within a single Node.js project, with environment-specific settings separated into a \texttt{.env} file.

\section{Implementation Environment}
\label{sec:impl-env}

\justifying
\noindent The project runs on Node.js (v20 LTS) with the Express.js framework. The database is a serverless PostgreSQL instance hosted on Neon, accessed through the \texttt{@neondatabase/serverless} driver over WebSockets, requiring no persistent TCP connection pool. Sensitive configuration values — database connection strings, JWT secrets, payment gateway keys, and Socket.io settings — are stored in a \texttt{.env} file at the project root and loaded via the \texttt{dotenv} package.

\section{Authentication Implementation}
\label{sec:auth}

\justifying
\noindent All authentication logic resides in \texttt{routes/auth.js}. Registration accepts a role parameter (\texttt{patient}, \texttt{doctor}, \texttt{admin}, or \texttt{lab}); passwords are hashed with \texttt{bcrypt} at a work factor of 12 before storage. On successful login, the server generates a signed JWT containing the user's ID, email, and role, written into an HTTP-only cookie with \texttt{Secure} and \texttt{SameSite=Strict} attributes to prevent client-side access and cross-site request forgery.

\justifying
\noindent Subsequent requests are authenticated by the \texttt{verifyToken} middleware, which reads the cookie, verifies the signature, and attaches the decoded payload to \texttt{req.user}. Role-specific guards (\texttt{requirePatient}, \texttt{requireDoctor}, \texttt{requireAdmin}, \texttt{requireLab}) wrap \texttt{verifyToken} and additionally compare \texttt{req.user.role} against the expected value, returning 403 on mismatch.

\justifying
\noindent Failed login attempts beyond a configurable threshold trigger a temporary account lockout, recorded against the user record, to limit the effectiveness of credential-stuffing attacks. The logout route clears the HTTP-only cookie and invalidates the server-side session entry in Redis, ensuring that a stolen cookie cannot be replayed after the user has signed out.

\section{Major Modules Implementation}
\label{sec:major-modules}

\justifying
\noindent This section presents the implementation of the four primary user-facing modules: Patient, Doctor, Admin, and Lab Technician. Each module follows consistent patterns for route protection, database interaction, and template rendering. The implementation details below cover the key workflows within each module.

\subsection{Patient Module Implementation}
\label{subsec:patient-impl}

\justifying
\noindent The patient module covers the full scope of functionality available to a registered patient. All routes are protected by the \texttt{requirePatient} guard. The module is organised within \texttt{routes/patient/} and split across several focused route files. All route handlers follow the same pattern: validation, parameterised database queries, and EJS template rendering.

\begin{itemize}
    \item \textbf{Patient Dashboard:} \\
    Rendered by \texttt{GET /patient/dashboard}, executing three parallel queries: upcoming appointments (joined with doctor name), recent prescription summary, and pending lab reports. Appointments are ordered ascending by date and time. Figure~\ref{fig:dashboard-patient} shows the rendered dashboard.

    \item \textbf{Appointment Booking Flow:} \\
    Proceeds through three sequential steps managed by \texttt{routes/patient/appointments.js}. Step 1 searches doctors with \texttt{ILIKE} filter. Step 2 fetches weekly availability, subtracts booked slots and leave periods. Step 3 opens a transaction, inserts appointment with status \texttt{pending\_payment}, and commits after payment. Figure~\ref{fig:appt-booking} shows the interface.

    \item \textbf{Prescription and Lab Report Viewing:} \\
    Prescriptions retrieved via join between \texttt{prescriptions}, \texttt{prescription\_items}, and \texttt{medicines}. Lab reports retrieved from \texttt{lab\_reports} filtered by \texttt{patient\_id}. Prescriptions display in reverse chronological order. Download links use signed URLs with short expiry windows.

    \item \textbf{Profile Management:} \\
    \texttt{POST /patient/profile} validates field lengths and formats, then issues \texttt{UPDATE} on \texttt{patients} table. Password changes require \texttt{bcrypt.compare} verification of current password. Dual validation applied client-side and server-side.
\end{itemize}

\label{subsec:doctor-impl}

\justifying
\noindent The doctor module provides clinicians with tools to manage daily schedules, conduct consultations, write prescriptions, review lab results, and request leave. Routes are protected by the \texttt{requireDoctor} guard and organised within \texttt{routes/doctor/}.

\begin{itemize}
    \item \textbf{Doctor Dashboard:} \\
    Rendered by \texttt{GET /doctor/dashboard}, fetching today's appointments ordered by time, count of pending lab reports, and a side-panel patient list for the current week. Designed to surface only current working day information. Figure~\ref{fig:dashboard-doctor} shows the dashboard.

    \item \textbf{Patient Consultation View:} \\
    \texttt{GET /doctor/appointment/:id} joins \texttt{appointments} with \texttt{patients} and loads previous prescriptions. Doctor adds notes to \texttt{appointments.doctor\_notes}. Route validates appointment ownership. Notes auto-save via background \texttt{PATCH} requests.

    \item \textbf{Prescription Writing:} \\
    \texttt{POST /doctor/prescription} inserts into \texttt{prescriptions} and iterates over medicine array inserting into \texttt{prescription\_items}. Both inserts run inside a single transaction. Medicine selection scoped to doctor's hospital.

    \item \textbf{Lab Report Review and Leave Application:} \\
    Reports retrieved from \texttt{lab\_reports} filtered by \texttt{doctor\_id}. Doctor marks reports as reviewed via \texttt{PATCH}. Reports sorted with unreviewed entries first. Leave form submits to \texttt{POST /doctor/leave} inserting into \texttt{doctor\_leaves} with status \texttt{pending}.
\end{itemize}

\subsection{Admin Module Implementation}
\label{subsec:admin-impl}

\justifying
\noindent The admin module gives hospital administrators oversight of facility operations including doctor and technician account management, medicine inventory, and leave approvals. Routes are protected by \texttt{requireAdmin} guard.

\begin{itemize}
    \item \textbf{Admin Dashboard:} \\
    Aggregates hospital-wide statistics through parallel \texttt{COUNT} and \texttt{SUM} queries: total registered patients, active doctors, current month appointments, and medicines below reorder threshold. Statistics refreshed on each load. Figure~\ref{fig:dashboard-admin} shows the dashboard.

    \item \textbf{Doctor Registration:} \\
    \texttt{POST /admin/doctors/create} hashes initial password, inserts into \texttt{users} with role \texttt{doctor}, then inserts profile into \texttt{doctors}. Both inserts wrapped in a transaction. Weekly availability stored as array in \texttt{doctors} table.

    \item \textbf{Medicine Inventory Management:} \\
    Lists all rows from \texttt{medicines} table with current stock and reorder level. Admins add medicines, update stock via parameterised \texttt{UPDATE}, and deactivate discontinued medicines with \texttt{is\_active} flag.

    \item \textbf{Lab Technician Account Creation and Leave Approval:} \\
    \texttt{POST /admin/lab/create} follows two-table pattern. Admin specifies department and shift. Pending leave requests viewed from \texttt{doctor\_leaves} where \texttt{status = 'pending'}. Approval updates status to \texttt{approved}, causing availability query to exclude those dates.
\end{itemize}

\subsection{Lab Technician Module Implementation}
\label{subsec:lab-impl}

\justifying
\noindent The lab technician module handles end-to-end laboratory report management. Routes are protected by \texttt{requireLab} guard. The two primary functional areas are the report submission form and the patient history table.

\begin{itemize}
    \item \textbf{Report Submission Form:} \\
    Rendered by \texttt{GET /lab/report/new} walking through structured multi-step process. Step 1 selects patient by name or ID. Step 2 selects test from pre-populated list and enters result value. Client-side validation flags out-of-range values. Step 3 configures sharing preferences (patient, doctor, or both). Figure~\ref{fig:lab-form} shows the interface.

    \item \textbf{Patient History Table:} \\
    Renders paginated table of previously submitted reports ordered descending by date. Each row shows patient name, test name, submission date, and sharing status. Edit link opens in-place pre-populated form for correction within configurable time window. Edits after window elapsed disabled at route level.
\end{itemize}

\section{Payment Implementation}
\label{sec:payment}

\justifying
\noindent Payment processing is integrated at \texttt{routes/payment.js}. When a patient confirms an appointment slot, the handler creates a payment intent via the gateway API using the fee from \texttt{doctors.consultation\_fee}. The returned client secret is passed to the EJS template, where client-side JavaScript mounts the payment widget without card details touching the application server.

\justifying
\noindent This architecture satisfies PCI DSS requirements by ensuring that sensitive card data is entered directly into a gateway-hosted iframe and never passes through the application's own network boundary. On successful authorisation, the gateway sends an asynchronous webhook \texttt{POST} to \texttt{/payment/webhook}. The handler verifies the event signature using the gateway's signing secret, then updates the \texttt{appointments} row from \texttt{pending\_payment} to \texttt{confirmed}. A payment record is also inserted into the \texttt{payments} table, capturing the gateway transaction reference, amount, and timestamp. If the webhook delivery fails or arrives out of order, the handler is idempotent.

\section{Real-Time Messaging Implementation}
\label{sec:messaging}

\justifying
\noindent Real-time chat is implemented with Socket.io attached to the Express HTTP server. When a user opens the messaging view, the client emits a \texttt{joinRoom} event carrying the appointment ID; the server places the socket into a named room derived from that ID, isolating messages per consultation. Room membership is validated server-side on every \texttt{joinRoom} event by confirming that the authenticated user is either the patient or the doctor associated with the given appointment.

\justifying
\noindent Incoming \texttt{sendMessage} events are persisted to the \texttt{messages} table — capturing sender ID, room ID, content, and timestamp — before broadcasting a \texttt{receiveMessage} event to the room. Persisting before broadcasting allows reconnecting users to retrieve full conversation history from the database on page load. Message content is sanitised server-side before storage to strip any HTML or script tags, preventing stored cross-site scripting. A Redis adapter shares the room registry across server instances, enabling horizontal scaling.

\section{Database Query Implementation}
\label{sec:db-queries}

\justifying
\noindent All database access goes through a thin helper in \texttt{db/index.js} exposing a single \texttt{query(sql, params)} function. Parameterised placeholders (\texttt{\$1}, \texttt{\$2}, \ldots) are used throughout to prevent SQL injection. The subsections below document the most significant query patterns employed across the application.

\subsection{Availability Check}

\justifying
\noindent The availability query returns only slots satisfying all three conditions simultaneously: the slot falls within the doctor's declared availability for that weekday, no confirmed appointment occupies it, and the date does not overlap with any approved leave. A \texttt{LEFT JOIN} from the generated slot series against \texttt{appointments} and \texttt{doctor\_leaves} filters where both join targets return \texttt{NULL}. The slot series is generated using a PostgreSQL \texttt{generate\_series} call bounded by the doctor's earliest and latest available times.

\subsection{Appointment Creation with Transaction}

\justifying
\noindent Appointment insertion is wrapped in an explicit transaction to guard against double-booking under concurrent requests. The transaction locks the target slot with \texttt{SELECT \ldots FOR UPDATE}, rechecks availability within the lock, inserts the appointment, and commits. If the recheck finds the slot taken, the transaction rolls back and the route returns a conflict response. The lock scope is intentionally narrow, targeting only the specific doctor and time slot combination.

\subsection{UUID Generation and Index Usage}

\justifying
\noindent Primary keys for \texttt{users}, \texttt{appointments}, \texttt{prescriptions}, and \texttt{lab\_reports} use UUIDs generated by PostgreSQL's \texttt{gen\_random\_uuid()} function as a column default, avoiding guessable sequential IDs in URLs. Indexes are defined on columns most frequently used in \texttt{WHERE} clauses: \texttt{appointments.patient\_id}, \texttt{appointments.doctor\_id}, \texttt{appointments.appointment\_date}, \texttt{lab\_reports.patient\_id}, and \texttt{doctor\_leaves.doctor\_id}. Query plans are verified during development using \texttt{EXPLAIN ANALYSE}.

\section{Summary}
\label{sec:implementation-summary}

\justifying
\noindent This chapter has presented the implementation of each module within the BondHealth platform. From the authentication layer through to the real-time messaging system, every component has been built with consistent emphasis on security, data integrity, and maintainability. The layered architecture separating routing, business logic, and database access ensures each module remains independently testable and extensible.

\noindent Parameterised queries, transactional inserts, and webhook-driven payment confirmation collectively reflect a production-grade approach to healthcare application development.
