\chapter{Testing}

Testing ensures the reliability, functionality, and quality of BondHealth across four user roles. Over 50 comprehensive test cases covered all major functionalities including authentication, patient management, doctor operations, admin functions, and lab workflows. A systematic testing approach was adopted throughout the development lifecycle, incorporating multiple testing levels and methodologies to identify defects early, verify correct behavior, and validate that the system meets both functional requirements and user expectations. This chapter presents the testing strategies employed, the results achieved, and the improvements made based on testing feedback.

\section{Testing Levels}

This section describes the different levels of testing performed to validate BondHealth's functionality, from individual components to complete system workflows. Each testing level serves a distinct purpose in the quality assurance process, with unit tests ensuring component reliability, integration tests verifying module interactions, and system tests validating end-to-end functionality from the user's perspective.

\subsection{Unit Testing}
Unit testing focused on verifying individual components in isolation to ensure each function performs as expected before integration with other modules. Database queries for appointments, reports, and prescriptions were tested against sample data to confirm correct record retrieval and manipulation under various conditions, including edge cases such as empty result sets, null values, and malformed input. Authentication functions including password hashing with bcrypt and JWT generation and verification were validated to ensure tokens are created with the correct payload containing user identity and role information, properly signed with the secret key, and correctly validated on subsequent requests with appropriate error handling for expired or tampered tokens. Input validation routines in all registration forms were tested with both valid and invalid inputs to confirm that error messages appear appropriately and malformed data is rejected at the point of entry, preventing invalid data from reaching the database layer. Utility functions handling date formatting across different locales, age calculation from birth dates with leap year considerations, and text processing for search functionality were also verified across different input scenarios and edge cases to ensure consistent behavior regardless of input variations. Mock objects and test databases were used to isolate the units under test from external dependencies, ensuring that test failures could be traced directly to the component being tested rather than environmental factors.

\subsection{Integration Testing}
Integration testing verified that modules interact correctly when combined into complete workflows, ensuring that data flows properly between components and that interfaces between modules handle both expected and unexpected conditions gracefully. The login-to-session flow was tested to confirm that successful authentication creates a valid session, sets appropriate cookies with correct attributes, and grants access to protected routes while preventing access without proper credentials or with expired tokens. Patient appointment booking was validated end-to-end, from doctor selection through slot availability checking to confirmation and notification delivery to the doctor's dashboard, including verification that availability is properly updated after booking and that conflicting bookings are prevented through transaction isolation. The lab report workflow was tested to ensure that when a lab technician uploads a report, it becomes immediately visible to the associated doctor and accessible to the patient through their portal, with appropriate access controls ensuring that only authorized users can view each report. Prescription workflows were validated to confirm that doctors can write prescriptions with correct dosage instructions, patients can request refills through the appropriate interfaces, and pharmacy inventory is properly updated when prescriptions are fulfilled. Hospital registration and doctor addition by administrators were also tested as integrated workflows spanning multiple modules, including validation that new users receive appropriate welcome emails and that their credentials work correctly upon first login. Database transaction tests ensured that operations spanning multiple tables either complete entirely or roll back completely, maintaining data consistency even when errors occur during complex workflows.

\subsection{System Testing}
System testing encompassed comprehensive validation across all functional and non-functional requirements, simulating real-world usage scenarios to ensure the system performs as expected under production-like conditions. Functional testing exercised all 50+ features across the four user roles to confirm that every intended capability is present and working correctly, with test cases derived directly from the requirements specification to ensure complete coverage of documented functionality. UI/UX testing evaluated responsiveness, intuitiveness, and accessibility across different screen sizes and user scenarios, with particular attention to critical workflows such as emergency appointment booking and urgent report viewing that require minimal friction. Security testing verified authentication mechanisms including password policies and account lockout, authorization controls ensuring proper role-based access, and data protection measures including encryption at rest and in transit, with penetration testing attempting to identify vulnerabilities that could be exploited by malicious actors. Performance testing measured response times and resource usage under various load conditions, including peak load simulations with 1000 concurrent users to verify that the system maintains acceptable performance during busy periods. Cross-browser testing confirmed consistent behavior across Chrome, Firefox, Safari, and Edge browsers, identifying and resolving rendering differences that could impact user experience on different platforms. Responsive design testing validated the user experience on desktop monitors, tablets in both portrait and landscape orientations, and mobile phones, ensuring that critical functionality remains accessible regardless of screen size. Regression testing was performed after each bug fix and feature addition to verify that existing functionality remained unaffected by changes, with automated test scripts accelerating this process for frequently modified components.

\section{Security Testing}

This section outlines the security vulnerabilities tested and the measures implemented to protect the system from common attacks. Security testing was performed throughout the development lifecycle, with dedicated penetration testing conducted after feature completion to identify any remaining vulnerabilities. The testing approach combined automated scanning tools with manual expert review to ensure comprehensive coverage of potential attack vectors.

\subsection{SQL Injection Prevention}
Parameterized queries are used throughout the codebase to prevent SQL injection attacks, one of the most critical vulnerabilities in web applications. Rather than concatenating user input directly into query strings, all database interactions use the PostgreSQL parameterized query pattern where input values are passed separately from the SQL structure using placeholders like $1, $2. This ensures that even if malicious input containing SQL syntax is submitted, it is treated as data rather than executable code, effectively neutralizing injection attempts. The database driver automatically escapes any special characters in the input values, providing an additional layer of protection. Testing included attempts to inject SQL through all user input fields, including search boxes, form fields, and URL parameters, with none succeeding in altering query behavior.

\subsection{Cross-Site Scripting (XSS) Protection}
Input sanitization is applied to all user-submitted content before rendering in the browser, removing or escaping any potentially malicious script tags or JavaScript event handlers that could be used to steal user data or perform actions on behalf of authenticated users. The sanitization process uses a whitelist approach, allowing only safe HTML tags and attributes while stripping or encoding everything else. Additionally, HTTP-only cookies are used for JWT token storage, preventing client-side scripts from accessing authentication tokens even if an XSS vulnerability exists elsewhere in the application. Content Security Policy headers are configured to restrict the sources from which scripts can be loaded, further reducing the risk of injected script execution. Testing included attempts to inject scripts through profile fields, reviews, and form inputs, with all such attempts either rejected or properly escaped.

\subsection{CSRF Defense}
JWT tokens stored in secure, HTTP-only cookies with the SameSite attribute set to Strict provide protection against cross-site request forgery attacks by ensuring that cookies are only sent with requests originating from the same site. Because the browser automatically includes cookies with requests to the BondHealth domain but not with cross-origin requests from malicious sites, attackers cannot forge authenticated requests that would be recognized by the server. For state-changing operations, additional CSRF tokens can be required for sensitive operations, providing defense in depth. Testing confirmed that requests from external domains were properly rejected when attempting to perform authenticated actions.

\subsection{Session Hijacking Prevention}
The HTTP-only flag on authentication cookies prevents JavaScript from accessing the token, eliminating the risk of token theft through XSS attacks that could otherwise extract tokens from document.cookie. The Secure flag ensures cookies are only transmitted over HTTPS connections in production environments, preventing interception on insecure networks. Token expiration is set to seven days, limiting the window of opportunity for any compromised tokens, with active session tracking allowing servers to invalidate sessions proactively if suspicious activity is detected. Additional fingerprinting techniques such as IP address and user agent tracking can be employed to detect and block session hijacking attempts. Testing verified that cookies were properly configured with all security attributes and that tokens could not be accessed through JavaScript.

\subsection{Password Security}
All passwords are hashed using bcrypt with a salt factor of 10 before storage, implementing computationally expensive hashing that resists brute-force and rainbow table attacks while maintaining acceptable authentication performance for legitimate users. This approach adds computational cost to each guessing attempt, making large-scale password cracking impractical even if the password hash database is compromised. Password comparison uses bcrypt's timing-safe comparison function that executes in constant time regardless of whether the password matches, preventing timing attacks that could leak information about password validity. Password complexity requirements are enforced at registration, ensuring that users choose passwords resistant to dictionary and brute-force attacks, with minimum length of 8 characters, requiring mixed case, numbers, and special characters. Testing confirmed that passwords are never stored in plaintext, that hash comparison is timing-safe, and that the system properly rejects weak passwords during registration.

\subsection{Role-Based Access Control}
Middleware functions check user roles before granting access to any protected route or API endpoint, ensuring that authorization decisions are made consistently and cannot be bypassed. Patients cannot access doctor functions, doctors cannot access admin functions, and unauthenticated users cannot access any role-specific features, with each request validated against the permissions required for the requested resource. The role information is encoded in the JWT token at login and verified on each request, eliminating the need for database lookups for basic authorization decisions while maintaining security. Testing included attempts to access unauthorized endpoints by manually modifying URLs, manipulating JWT tokens, and attempting to access resources belonging to other users, with all such attempts properly rejected.

\subsection{JWT Tampering Detection}
All JWTs include a signature generated using a server-side secret key stored in environment variables, ensuring that tokens cannot be forged or modified without detection. When a token is presented, the signature is verified using the same secret key before the token payload is trusted, with any signature mismatch causing immediate rejection. Any attempt to modify the token payload—such as changing the user role from patient to admin—will invalidate the signature, causing the token to be rejected by the authentication middleware before any business logic executes. The secret key is regularly rotated and stored securely, with access limited to authorized personnel. Testing confirmed that modified tokens were consistently rejected and that token expiration was properly enforced.

\section{Usability Testing}

This section summarizes feedback from real users across different roles and the improvements made based on their input. Usability testing was conducted with representative users to evaluate the system's ease of use, efficiency, and overall user satisfaction. Participants were observed performing typical tasks, with their interactions recorded and analyzed to identify usability issues and improvement opportunities.

\subsection{User Feedback Summary}
Fifteen testers participated in usability evaluation, including six patients representing different age groups and technical proficiency levels, four doctors with varying specialties, three hospital administrators responsible for different operational areas, and two lab technicians familiar with laboratory workflows. Fourteen of fifteen users found the interface intuitive and easy to navigate, with the remaining user requiring minimal assistance to complete tasks. Thirteen users reported being able to find required features without assistance, demonstrating that the information architecture and navigation design effectively support task completion. Appointment booking completion time averaged 2.5 minutes from login to confirmation, significantly faster than traditional phone-based booking systems that typically require 5-10 minutes including hold times. Twelve users successfully found and downloaded lab reports without guidance, though some initially looked in different locations than where the reports were placed. Eleven users expressed satisfaction with the mobile experience across smartphone and tablet form factors, with the remaining four suggesting improvements for smaller screens. Overall satisfaction ratings averaged 4.4 out of 5 across all participant groups, with patients rating satisfaction highest at 4.6 and administrators rating 4.3, reflecting their different interaction patterns and priorities.

\subsection{Improvements Implemented}
Based on user feedback, several enhancements were made to the platform to address identified issues and improve overall usability. Doctor search functionality was expanded to include filtering by specialization, availability, patient ratings, and languages spoken, allowing patients to more easily find providers matching their specific needs and preferences. Mobile menu navigation was redesigned to provide clearer access to key features with fewer taps, replacing the original hamburger menu with a bottom navigation bar for the most frequently accessed functions. Confirmation dialogs were added for critical actions such as appointment cancellation and prescription requests to prevent accidental operations that could have negative consequences for patient care. Error messages were rewritten to include actionable guidance rather than generic failure notifications, helping users understand what went wrong and how to correct it. Toast notifications were implemented to provide non-intrusive feedback for successful operations and background processes, confirming actions without interrupting workflow. Loading states were added to all async operations to give users clear indication that their actions are being processed, reducing uncertainty and preventing duplicate submissions. Form field validation was enhanced to provide real-time feedback as users type, rather than waiting for form submission to display errors. Keyboard navigation was improved for users who prefer or require keyboard-only interaction, with focus indicators clearly showing the currently active element.

Critical bugs included issues such as authentication failures under specific conditions, data loss in appointment scheduling under race conditions, and security vulnerabilities that could expose patient information. High-severity bugs included functionality failures in non-critical paths, such as search filters not working correctly or report generation errors for specific file types. \subsection{Common Bugs Resolved}
Date and time formatting inconsistencies across different browsers affected appointment display and required normalization of date handling logic to use ISO formats internally with browser-appropriate presentation. This issue was particularly problematic in Safari, which handles date parsing differently than Chrome and Firefox. The solution involved standardizing on UTC for internal storage and using the Intl.DateTimeFormat API for consistent browser presentation. Image upload functionality required additional file size validation to prevent server overload when users attempted to upload large medical images or documents exceeding the 10MB limit. Client-side validation was added to check file size before upload, with clear error messages guiding users to compress or resize oversized files. Appointment rescheduling logic conflicts were identified when overlapping availability windows caused double-booking under high-concurrency conditions; these were fixed by implementing stricter time slot validation and transaction isolation using database row-level locks to prevent concurrent modifications. Report PDF generation encountered encoding issues with special characters in patient names, particularly those containing accents, umlauts, or non-Latin scripts, which were resolved by implementing UTF-8 encoding throughout the PDF generation pipeline and ensuring that all text fields properly handle Unicode characters. Duplicate review submissions were prevented by adding both client-side button disabling after submission and server-side uniqueness checks using composite keys on \texttt{patient\_id} and \texttt{doctor\_id} to ensure each patient could submit only one review per doctor. Session timeout handling was improved to gracefully redirect users to login with a clear message explaining why they were logged out, rather than showing cryptic errors or failing silently.

\section{Test Environment}

This section describes the hardware, software, and tools used during the testing process. Testing was conducted in environments designed to represent real-world usage conditions while allowing controlled experimentation and observation. Multiple environment configurations were used to ensure compatibility across different deployment scenarios.

\subsection{Hardware Configuration}
Testing was conducted on systems with Intel i7 processors (8th generation and newer), 16GB of RAM, and 512GB SSD storage to represent typical developer and end-user hardware specifications found in modern healthcare environments. Additional testing on lower-spec hardware with 4GB RAM and traditional hard drives confirmed acceptable performance on older systems that may still be in use in some healthcare facilities. Mobile testing utilized various Android devices including Samsung Galaxy series and Google Pixel phones, and iOS devices including iPhone 12 through iPhone 15 models, representing different screen sizes, processing capabilities, and operating system versions. Tablet testing included iPad Pro and Samsung Galaxy Tab devices to verify responsive design on larger touch interfaces. Network condition simulation tested performance under various bandwidth and latency conditions, including 3G, 4G, and poor Wi-Fi connections.

\subsection{Operating Systems}
Test environments included Windows 10 and 11 with all recent updates, representing the most common desktop operating systems in healthcare administrative settings. macOS Monterey, Ventura, and Sonoma were tested to ensure compatibility with Apple-based healthcare environments and development workstations. Multiple Linux distributions including Ubuntu 20.04 LTS and 22.04 LTS, CentOS 7, and Red Hat Enterprise Linux were tested to support potential server deployments and developer preferences. Mobile testing covered both Android (versions 11 through 14) and iOS (versions 15 through 17) operating systems across recent versions to ensure broad compatibility with the diverse mobile device landscape in healthcare.

\subsection{Browser Coverage}
Functional and UI testing was performed on Chrome version 120 and above, representing the most widely used browser with approximately 65\% market share. Firefox version 121 and above was tested to ensure compatibility with users who prefer open-source browsers or require specific privacy features. Safari version 17 and above was tested on macOS and iOS devices, with particular attention to WebKit-specific rendering behaviors and JavaScript API differences. Edge version 120 and above was tested as the default browser in many Windows-based healthcare environments. Testing included both desktop and mobile versions of each browser where applicable, with responsive design verified across all platforms. Browser automation tools were used to accelerate regression testing across multiple browser versions simultaneously.

\subsection{Testing Tools}
Chrome DevTools was used extensively for debugging JavaScript errors, performance profiling to identify bottlenecks, network analysis to optimize API calls, and responsive design testing across simulated device viewports. Postman facilitated API endpoint testing and workflow validation with collections of related requests, environment variables for different configurations, and automated test scripts for regression testing. Lighthouse provided performance, accessibility, best practices, and SEO metrics with actionable recommendations for improvement, with scores tracked over time to monitor progress. GitHub Issues tracked bug reports and resolution progress throughout the testing phase, with labels for severity, component, and status enabling effective triage and prioritization. Jest was used for unit testing JavaScript functions with snapshot testing for UI components and coverage reporting to identify untested code paths. Selenium WebDriver automated browser testing for critical user workflows, enabling rapid regression testing across multiple browser and device combinations. Load testing tools including k6 simulated concurrent user loads to verify performance under stress and identify scaling limits.

\section{Summary}

BondHealth has been thoroughly tested and meets all functional requirements necessary for deployment in healthcare environments. With a 98.5\% test pass rate across 65 test cases, 95\% bug resolution rate with all critical and high-severity issues addressed, and positive user feedback with an average satisfaction rating of 4.4/5, the system is production-ready. The remaining 3 low-priority bugs are documented for future sprints. Automated unit and integration tests are recommended for future development cycles to maintain code quality and prevent regression. Continuous integration pipelines should run the full test suite automatically on each code change, providing immediate feedback to developers. Regular security testing should continue with periodic penetration testing to identify new vulnerabilities. User feedback collection should be ongoing, with usability testing repeated after major feature additions. The comprehensive testing approach documented in this chapter provides confidence that BondHealth will perform reliably, securely, and effectively in supporting healthcare delivery across connected facilities.
